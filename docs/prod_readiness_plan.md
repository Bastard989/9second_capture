# 9second_capture: план готовности к production

Дата ревизии: 2026-03-05

## 1. Целевое поведение (текущая архитектура)

- Запись/импорт MP3 выполняются отдельно от обработки.
- Транскрипция STT запускается только вручную пользователем.
- LLM-генерация запускается только вручную пользователем по прикреплённым файлам.
- UI работает в трех режимах: browser, api/file, link fallback.

## 2. Что уже должно быть зафиксировано

- Нет обязательной зависимости от старого режима системного драйвера в пользовательском потоке.
- Нет автозапуска транскрипта во время записи.
- LLM и embeddings модели разделены по отдельным селекторам.
- Результаты генерации/транскрипции отображаются через единый busy-overlay с прогрессом.

## 3. P0 до релиза

1. Пройти e2e smoke по ключевому сценарию:
- запись -> стоп -> сохранить MP3 -> ручной STT -> ручной LLM.
2. Проверить обработку больших MP3 (длинные встречи).
3. Проверить корректную отмену (`Прервать`) долгих операций.
4. Подтвердить устойчивость при недоступности LLM провайдера.

## 4. P1 улучшения

1. Добавить метрики времени по этапам:
- stop -> mp3 ready
- transcript request -> raw ready
- llm request -> result ready
2. Ввести верхние guardrail-пороги для предупреждений UX (не прерывая операцию).
3. Автоматизировать regression smoke в CI для основных API роутов.

## 5. Definition of Done

- Пользовательский сценарий проходит без ручного вмешательства в конфиги.
- Документация соответствует фактическому UI и API.
- Нет ссылок на старую логику режима системного драйвера в пользовательских гайдах.
- Unit/smoke проверки зелёные.
