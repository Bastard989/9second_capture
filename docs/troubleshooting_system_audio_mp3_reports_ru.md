# 9second_capture: проблемы MP3 / STT / LLM (актуально)

Документ для текущего агента: запись браузерного аудио, ручной STT, ручной LLM.

## 1. Долгий импорт MP3

Почему долго:
- файл копируется во внутреннее хранилище агента;
- обновляется индекс записей;
- для больших файлов время растет линейно.

Что проверить:
1. Размер исходного файла.
2. Скорость диска (лучше локальный SSD).
3. Отсутствие параллельной heavy-нагрузки в системе.

## 2. Долгая транскрипция

Главные причины:
- тяжелая STT модель (`medium`/`large-v3`);
- длинная запись;
- CPU-only инференс.

Как ускорить:
1. В `Обработка -> STT модель` выбрать `small`.
2. Запускать только нужный вариант (`Raw` или `Clean`).
3. Закрыть лишние тяжелые приложения на время STT.

## 3. Плохое качество транскрипта

Причины:
- не включен `Share audio` при старте записи;
- шумный микрофон и низкий уровень системного звука;
- неподходящий языковой профиль.

Решение:
1. Перезапустить короткую запись, обязательно с `Share audio`.
2. В `Обработка` выбрать корректный языковой профиль (`mixed/ru/en`).
3. Для качества повысить STT модель до `medium`.

## 4. LLM fallback вместо результата

Симптом:
- `llm_unavailable_fallback`

Проверка:
1. В `LLM модель` выбрать chat-модель (например `llama3.1:8b`).
2. В `Embeddings модель` выбрать embedding-модель отдельно.
3. Проверить доступность выбранного LLM-провайдера и его API URL.

## 5. Пустая/плохая таблица из LLM

Причины:
- приложен не тот файл;
- файл слишком "грязный" (один длинный абзац без сегментов);
- промпт просит структуру, но в тексте нет явных owner/due_date и т.д.

Практика:
1. Давайте LLM `clean`-текст или предварительно сегментированный TXT.
2. В промпте явно указывайте: не выдумывать факты, ставить `Не указан`.
3. Для длинных встреч разбивайте вход на части и объединяйте результат.

## 6. MP3 не сохраняется

Проверьте:
1. Выбрана запись в списке.
2. Выбрана папка для сохранения.
3. Для ручного запуска без .app установлен `ffmpeg`.

## 7. Полезные команды

```bash
cd <project-directory>

```

```bash

```
